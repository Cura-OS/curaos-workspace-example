User vision (2026-05-24 interview, ADR-0100 cycle):

## Core architecture metaphor: Injection Molding

1. **Build the MOLD first** = the foundational platform pieces:
   - **Auth/IdP** (standalone SaaS product)
   - **App/Site Builder** (standalone SaaS product)
   - **Workflow Manager** (standalone SaaS product)
   - … plus other foundation services that are sellable on their own AND compose

2. **Use the mold to PRODUCE PARTS** = downstream services, business modules, vertical overlays. If the foundation is strong enough, downstream services need minimal manual coding (human or AI).

3. **Manually craft** only specialized services where the stack can't generate.

## Each service is its own product

- Every CuraOS service must be sellable standalone (independent SaaS, on-prem, hybrid).
- Composed together = full platform (modulith or microservice system).
- Building-block economics: customers can buy 1 service, then add more.
- Implications: each service has its own pricing, packaging, branding hooks, install path, docs, integration boundary.

## Stable core + plugin/sidecar interceptor pattern (NOT hot-reload)

- **Core service code = stable.** Compiled. Rolling Helm deploy only for upgrades/security.
- **Plugins + sidecars** = the runtime extension mechanism.
  - Code-generated by CuraOS platform (NOT same path as core codegen)
  - Inject extra business logic, requirements, scripts, sanitizations
  - Connect via **event interceptors at runtime** (the events flowing through ADR-0102 stream/queue layer become the injection point)
- **Pattern preferred over full hot-reload.** Plugins/sidecars carry tenant-specific + business-rule changes.
- **BPMN: fallback only.** ADR-0122 (`0122-foundation-workflow-manager.md`) is the workflow-paradigm authority (Temporal + Activepieces + BullMQ). See RESOLUTION-MAP.md for current status. BPMN remains fallback; primary no-code/low-code paradigm is per ADR-0122. (ADR-0105 is superseded by ADR-0122.)

## What gets generated vs what gets coded

- **Generated by platform (codegen):**
  - Core service scaffold (per chosen runtime, OSS-bridged)
  - Plugins/sidecars per tenant customization
  - API/types/event/DB-binding bindings (forward + backward engineering)
  - Event-interceptor wiring
- **Manually coded:**
  - Foundation platform (Auth, Builder, Workflow Manager initially)
  - Specialized services that can't be molded
  - Algorithmic core where DSL would be wrong tool (e.g., medical imaging codec, specific clinical algorithms)

## Build order

1. Foundation: Auth + App/Site Builder + Workflow Manager (sellable standalone, become injection mold).
2. Generic building-block services (using the mold).
3. Vertical overlays (HealthStack/EducationStack/ERP) layered as "big plugins" or specialized services.
4. Only after foundation works, evaluate whether specialized services are needed or stack can handle.

## Behavior reminders

- BPMN: fallback only; primary workflow/BPM paradigm resolved in ADR-0122 (Temporal + Activepieces + BullMQ).
- New ADR recommendations stay tentative until user explicitly confirms via funnel (per [[curaos-decision-methodology]]). Settled ADRs (0100, 0101, 0122, 0123) are not reopenable without a new superseding ADR.
- Each ADR must consider: does this support each-service-is-a-product economics?
- Stable-core + plugin/sidecar/interceptor preferred over hot-reload of core itself.

## Codegen sourcing decision

- Combined approach: **(option 2) bridge existing OSS + (option 3) fork a comprehensive scaffolder**.
- Examples to evaluate: OpenAPI Generator + Atlas + sqlc + Protobuf/Buf + Hasura/PostgREST.
- Comprehensive scaffolder candidates: Encore.dev, Buffalo, JHipster (Kotlin if JVM chosen), or build from Encore-style template.
- **Critical scalability check:** PostgREST + similar "auto-API-from-DB" tools must scale to big-SaaS load. Research multi-tenant + high-QPS limits.
- **DB choice:** Resolved in ADR-0101 (`0101-data-layer.md`). See `ai/curaos/docs/adr/RESOLUTION-MAP.md` for current status.
