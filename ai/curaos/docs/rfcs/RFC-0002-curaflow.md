## Summary
CuraFlow is the workflow and BPM engine that powers orchestration across CuraOS. It delivers BPMN-equivalent modeling, execution, and human task management without inheriting XML baggage. CuraFlow introduces the **CuraFlow Definition Language (CFDL)** — a structured JSON specification designed for Git-friendly diffs, schema validation, and code generation. The engine deploys inside `backend/services/workflow-core-service`, feeds personal/business/HealthStack overlays, and drives Flutter task surfaces (web, desktop, mobile) composed from reusable builder components.

## Problem Statement
Current BPMN implementations rely on XML representations and external designers that do not align with CuraOS design guardrails. They complicate version control, dev tooling, and integration with Flutter task surfaces and the site/app builder. CuraOS needs a native workflow engine that:
- Models complex processes, gateways, timers, and compensation the way BPMN 2.0 does.
- Stores definitions in a developer-friendly, versioned format.
- Produces human task assignments that render as Flutter pages across device targets.
- Coordinates with the builder (future RFC) without circular dependencies.

## Goals
- Ship a BPMN-equivalent execution engine in `workflow-core-service` with tenant isolation, event sourcing, and APIs consistent with the existing service charter.
- Define CFDL (JSON + JSON Schema) as the canonical format for modeling, validation, and code generation; support import/export bridges for BPMN XML when interoperability is required.
- Generate Flutter task surfaces from reusable task templates that can be embedded in hosted apps (`frontend/curaos-apps/packages/cura_os/personal_workflow`, `.../business_workflow`, HealthStack variants).
- Provide a diagramming/editor experience tailored to Flutter developers while keeping the engine headless and embeddable.
- Minimize dependency loops with the site/app builder by sharing contracts (schemas, component registries) via lightweight packages rather than cross-service calls.

## Non-Goals
- Re-implementing traditional BPMN XML authoring tools or forcing XML as the source of truth.
- Owning domain-specific service adapters; vertical overlays will extend service tasks.
- Building a WYSIWYG app/builder experience in this RFC (covered in future RFC for Builder).

## Architecture
| Component | Responsibility | Repo/Path |

| Observability | Metrics, traces, audit streams (`workflow.instances.v1`, `workflow.audit.v1`) | Kafka + OTLP |

## CuraFlow Definition Language (CFDL)
- **Format:** JSON documents validated against versioned JSON Schemas (`docs/specs/cfdl/v1/schema.json`, to be generated by `make docs`).
- **Top-level structure:**
  ```json
  {
    "$schema": "https://schemas.curaos.dev/cfdl/v1.json",
    "id": "order-fulfillment",
    "version": "1.3.0",
    "metadata": {
      "name": "Order Fulfillment",
      "description": "Handles intake, verification, and fulfillment for orders.",
      "tags": ["commerce", "sla:48h"],
      "tenantScope": "shared"
    },
    "context": {
      "inputs": {...},
      "outputs": {...},
      "variables": {...}
    },
    "graph": {
      "nodes": [...],
      "edges": [...]
    },
    "forms": [...],
    "policies": {...},
    "extensions": {...}
  }
  ```
- **Nodes:** typed objects (`start.event`, `exclusive.gateway`, `parallel.gateway`, `user.task`, `service.task`, `script.task`, `timer.event`, `compensation.event`, `subprocess`, `end.event`). Each node declares:
  - `id`, `type`, `name`, `lane`, `metadata`
  - `ioBindings`: describes variable mappings
  - `conditions`: for gateways or boundary events
  - `taskRef`: references to task templates or service adapters
- **Edges:** adjacency definitions with optional guards (`expression`, `priority`, `default`).
- **Forms:** attach UI/task definitions referencing Flutter components (see Task Templates).
- **Policies:** scheduling (SLAs, retry), security (required roles/attributes), multi-instance semantics.
- **Extensions:** plugin slots for overlays (e.g., HealthStack compliance metadata) stored under namespaced keys (`"com.cura.healthstack": {...}`).
- **Interoperability:** Provide CLI import/export handlers to translate between BPMN XML and CFDL for partners; CFDL remains the authoritative source.

## Task Templates & Flutter Pages
- **Task Templates:** JSON structures referencing component specifications (FormKit) consumed by Flutter packages. Each template maps to Dart widgets within `frontend/curaos-apps/packages/cura_os/ui_kit`.
- **Renderer:** Workflow packages (`personal_workflow`, `business_workflow`) interpret templates, fetch task payloads via GraphQL (persisted queries), and render responsive layouts for mobile/web/desktop using adaptive scaffolds.
- **Config Storage:** Task templates are stored alongside CFDL definitions and cached per tenant. Builder tooling can update templates without redeploying the engine.
- **Navigation:** Completed tasks emit events to the engine via WebSocket/GraphQL mutation and optionally notify the builder for UI state updates.

## Editor Strategy
- **Primary Path:** Evolve `frontend/curaos-apps/packages/cura_os/workflow_designer`, a Flutter-based diagramming/editor package assembled from:
  - `diagram_editor` (open-source graph editor for Flutter) as the canvas foundation.
  - Custom palettes, property inspectors, and validation UI matching CFDL semantics.
  - CFDL schema-aware code completion, validation, and diff previews (leveraging Monaco via Flutter webview for advanced editing when needed).
- **Interim Workflow:** Developers author CFDL in code editors (JSON with schema validation) while the Flutter designer matures. Provide CLI (`make cfdl-validate`) for schema + lint.
- **Rationale:** Existing low-code BPMN designers either enforce XML or lack Flutter compatibility. Investing in a Flutter package keeps the modeling experience close to our UI stack, enabling reuse in admin apps and the future site/app builder.

## APIs & Integration
- **Definition APIs:** CRUD, version lifecycle (draft → active → retired), diff, copy-as template.
- **Execution APIs:** Start process, signal event, manage timers, query instance history, fetch audit trails.
- **Task APIs:** Inbox listing, claim/complete, escalate/delegate, deliver form configs, attach files (via MinIO).
- **Events:** Kafka topics (`workflow.instances.v1`, `workflow.tasks.v1`, `workflow.audit.v1`, `workflow.metrics.v1`) consumed by reporting, automation, and builder telemetry.
- **SDKs:** Kotlin (backend services), Dart (Flutter apps), JS/TS (builder web tooling), Go (ops automation). SDKs share CFDL schema definitions.
- **Policies:** Leverage identity-service for roles/attributes and tenancy. Ensure RBAC + ABAC enforcement during task fetch/complete operations.

## Deployment Profiles
- **Local:** Docker Compose with Postgres, Redis, Kafka, MinIO; sample CFDL definitions seeded via Make task.
- **On-Prem:** Helm chart (`ops/helm/curaflow`, new) with sealed secrets, horizontally scalable workers, optional external scheduler.
- **SaaS:** Multi-regional deployment with partitioned Kafka topics, externalized storage, and per-tenant encryption keys.

## Dependencies & Interplay
- **identity-service:** Used for user resolution, roles, tenant scoping.
- **notify-service:** Subscription channel for task notifications (email, push, in-app).
- **event-core-service:** Central event routing; ensures builder can react to workflow state.
- **builder (RFC-0003 forthcoming):** Shares component registry and layout metadata. Avoid direct runtime coupling by exchanging versioned packages and schemas.
- **storage-service/minio:** Attachment handling for tasks.

## Impact
- **Backend:** Centralizes workflow orchestration, enabling other microservices to offload complex process logic.
- **Frontend:** Supplies consistent task UX across personal/business/HealthStack Flutter apps with minimal duplication.
- **Ops:** Adds new helm chart, scaling policies, and observability for long-running workflow instances.
- **Docs/Tooling:** Introduces CFDL schema, CLI validation, designer package documentation, and codex updates for workflow services.
