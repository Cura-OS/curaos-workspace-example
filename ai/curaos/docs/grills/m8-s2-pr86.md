**P1-02 — digest extraction multi-arch: RESOLVED**

Evidence: `tools/build/repro-build.sh` `oci_tar_digest()` function (lines ~195-240 in patch):
- Extracts `index.json` from OCI tarball via `tar -xOf`
- Uses `jq` with `--arg os` + `--arg arch` to filter by `.platform.os == $os and .platform.architecture == $arch`
- Explicitly filters by `mediaType` accepting only `oci.image.manifest`, `oci.image.index`, `docker.distribution.manifest`
- Explicitly excludes attestation descriptors: `(.annotations["vnd.docker.reference.type"] // "") != "attestation-manifest"`
- Returns `.[0].digest` from the FILTERED list (not raw `manifests[0]`)
- Script rejects comma-separated `--platform` values, enforcing single-platform per invocation

### Stale-reference scan

### Per-checklist verdict

1. **Same-tool determinism** — `tools/build/repro-build.sh` sets `SOURCE_DATE_EPOCH`, threads it through both BuildKit and Buildah invocations, disables BuildKit provenance/SBOM attestations, uses `--rewrite-timestamp`, and selects platform-aware OCI manifest descriptors. No random or time-based values written into layers. Determinism posture is sound.

2. **Dockerfile fixture** — `.github/workflows/repro-build-test-images/Dockerfile` exercises: digest-pinned base image, two `bun install` stages, source `COPY`, `bun build` output, multi-stage `COPY --from=`, and final `find … touch` normalization. All relevant timestamp sources covered.

3. **Workflow gates** — `.github/workflows/repro-build.yml` runs four same-tool jobs (`bk-determinism` and `ba-determinism` across `linux/amd64` and `linux/arm64`), applies path-filtered PR/main triggers, cancels superseded runs with `cancel-in-progress: true`, and uploads per-pass artifacts on failure. 30-minute timeout is sufficient for the fixture workload. Gates are correct.

4. **README rationale** — `tools/build/README.md` §"Why cross-tool equivalence is NOT the contract" argument holds. The Dangerzone citation is accurate (Freedom of the Press Foundation gates same-tool reproducibility; cross-tool divergence is documented as expected due to metadata differences). The claim that same-tool determinism is "sufficient for cosign" is correct: cosign signs and verifies a specific digest, so what matters is that the same builder always produces the same digest on the same input — cross-tool equivalence is irrelevant to the signing contract.

5. **Stale references** — None found (see scan above).

6. **ADR adequacy** — **ADR is warranted and must resolve before merge.** Neither ADR-0109 (containers tooling) nor ADR-0110 (CI/CD releases) covers this specific decision. ADR-0164 (Zarf bundle layout) currently preserves the old cross-tool-identical claim at lines 115-117. The authorized v4 pivot — cross-tool gating dropped after three failed approaches, same-tool determinism as the release gate, expected digest divergence between BuildKit and Buildah documented — is a material architectural decision that needs a formal record. Options: (a) add a new ADR-NNNN for the reproducible-build contract, or (b) add a resolution-pin banner to ADR-0164 lines 115-117 pointing to the authorized decision. Either satisfies the P1 condition.

### Summary

The v4 implementation is technically sound: the repro-build script eliminates nondeterminism sources, the Dockerfile fixture covers all timestamp touchpoints, the four-job workflow is correctly structured, and no stale cross-tool references remain in the codebase. The single merge condition is the stale cross-tool-equivalence claim in ADR-0164, which contradicts the authorized v4 contract and must be corrected or superseded by a new ADR before this PR lands.
