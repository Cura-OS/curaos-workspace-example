## P0 findings (block merge)

### 1. Base image tag-only pin — supply-chain vulnerability + air-gap non-determinism

- **Where:** `ops/migrations/migration-runner.Dockerfile:24`
- **What:** `FROM oven/bun:${BUN_VERSION}-alpine AS base` with `ARG BUN_VERSION=1.3.14`. Tag-only; no `@sha256:<digest>`. The tag `1.3.14-alpine` is a mutable pointer on Docker Hub — it can be retargeted to a different layer tree at any time.
- **Why it's P0:** `curaos_version_pinning_rule.md` line 24 states the mandatory pattern as `oven/bun:1.x@sha256:...` with cosign verification. A tag-only base image means: (a) two CI builds at different wall-clock times can produce non-identical migrator images even with the same `BUN_VERSION` arg — breaking Zarf bundle reproducibility (`[[curaos-airgap-rule]]`); (b) a supply-chain compromise (Docker Hub tag hijack) yields a malicious image that runs with the DB credentials of every CuraOS service; (c) `docker:pinDigests` Renovate config is already present in the workspace `renovate.json` — the PR simply doesn't comply.
- **Fix:** Replace line 24 with:
  ```dockerfile
  FROM oven/bun:1.3.14-alpine@sha256:<digest-from-docker-hub-manifest>
  ```
  Add a Renovate `matchManagers: ["dockerfile"]` entry to keep it fresh. Also add: per-service overlay `README.md` note that their `Dockerfile.migrator` `ARG BASE_VERSION` must reference an already-digest-pinned tag in the registry.

### 2. No cross-Job pg_advisory_lock — concurrent migrations corrupt Drizzle metadata

- **Where:** `ops/migrations/run-migrations.ts:51-52` — no advisory lock acquired before `migrate(db, { migrationsFolder })`.
- **What:** `drizzle-orm/node-postgres/migrator`'s `migrate()` does NOT acquire a cluster-wide `pg_advisory_lock` before reading `__drizzle_migrations`. Two Job pods running concurrently (possible: `helm upgrade` issued twice by an operator in rapid succession, or a pre-upgrade hook fires while a prior pre-install pod is still in its backoffLimit retry) will both read the current applied migrations list, decide the same pending migrations need to run, and attempt to apply them simultaneously. Depending on DDL statement idempotency, this produces duplicate-constraint errors or — worse — a torn `__drizzle_migrations` row.
- **Why it's P0:** `job-template.yaml:53` sets `parallelism: 1` and `completions: 1`, which prevents parallel Pods within a *single* Job. But it does NOT prevent two independent Jobs from running concurrently — e.g., `helm upgrade` fires a new pre-upgrade Job while the previous Job's retry (backoffLimit not yet exhausted) is still running under a different revision number (`{{ .Release.Revision }}`). The naming scheme `{{ .Values.service.name }}-migrator-{{ .Release.Revision }}` creates a new Job name per revision, so Kubernetes does not detect a conflict. The LOCK_TIMEOUT_MS (30 s) only guards DDL lock waits, not serialization of the migration sequence itself. No `SELECT pg_advisory_lock(...)` call exists anywhere in the diff.
- **Fix:** Before calling `migrate()`, acquire an advisory lock keyed to the service name:
  ```typescript
  const lockKey = hashServiceName(process.env.SERVICE_NAME ?? 'curaos-migrator');
  await pool.query('SELECT pg_advisory_lock($1)', [lockKey]);
  try {
    await migrate(db, { migrationsFolder: MIGRATIONS_FOLDER });
  } finally {
    await pool.query('SELECT pg_advisory_unlock($1)', [lockKey]);
  }
  ```
  Alternatively: add a `helm.sh/hook-weight` gate that prevents a new Job from starting until all same-weight Jobs for the same service name have exited.

### 3. `restartPolicy: OnFailure` + no transactional wrapper = partial-migration replay on retry

- **Where:** `ops/migrations/job-template.yaml:62` (`restartPolicy: OnFailure`); `ops/migrations/run-migrations.ts:51-52` (`migrate(db, ...)`).
- **What:** Drizzle's `migrate()` applies each migration file in a *separate* implicit transaction (one `BEGIN`/`COMMIT` per file). If the Pod is preempted mid-migration (OOM, node failure, SIGTERM from `activeDeadlineSeconds:900`), the current migration file's transaction is rolled back, but the `__drizzle_migrations` row for that file was not yet inserted — so the file-level idempotency gate (Drizzle checks the table before applying each file) correctly re-applies it on retry. **However:** the forward-only-policy.md rule 6 states "statement-level idempotency protects against partial replays after pod preemption mid-migration." This is only true if every SQL statement in every migration uses `IF NOT EXISTS` / `ON CONFLICT DO NOTHING`. The PR does not ship any CI gate that enforces statement-level idempotency in per-service migration files. A single `CREATE TABLE foo (...)` (without `IF NOT EXISTS`) in any service migration will produce a fatal error on retry, exhausting backoffLimit and leaving the cluster in a partially-migrated state that requires manual PITR recovery.
- **Why it's P0:** The gap is not in the runner itself but in the enforcement mechanism: the CI gate that validates statement-level idempotency (`bun run scripts/check-migrations.ts`) is mentioned in `forward-only-policy.md:25` but **the script does not exist in this PR's diff** and no CI workflow step was added. The policy is stated but unenforced. Any service author shipping `CREATE TABLE` without `IF NOT EXISTS` will silently bypass the contract, hit a retry scenario in production, and exhaust the Job's backoffLimit.
- **Fix:** Ship `scripts/check-migrations.ts` in this PR (or as an immediate follow-up explicitly tracked and linked). Add a step to the Tier-B CI workflow (`.github/workflows/tier-b-fast-ci.yml`) that runs `bun run ops/migrations/scripts/check-migrations.ts --dir $SERVICE_MIGRATIONS_DIR` per service. Until the script exists, the CI gate claim in `forward-only-policy.md:25` is false advertising.

## P1 findings (must address before merge)

### 4. `hook-delete-policy: before-hook-creation` only — failed Jobs not cleaned up on success

- **Where:** `ops/migrations/job-template.yaml:44`
- **What:** `"helm.sh/hook-delete-policy": before-hook-creation` deletes the *previous* Job when a new hook creation is triggered. It does NOT delete the current successful Job. The PR relies on `ttlSecondsAfterFinished: 3600` (line 50) for GC of successful Jobs. This is correct for happy-path, but `ttlSecondsAfterFinished` requires the `TTLAfterFinished` feature gate, which is beta-on by default in K8s 1.21+ but may be disabled in some K3s hardened profiles. If TTL GC is disabled, successful Jobs accumulate indefinitely. Additionally, `before-hook-creation` does not clean up Jobs that are still running when a new upgrade fires — which is part of the concurrent-migration scenario in P0-2.
- **Fix:** Add `hook-succeeded` to the delete policy: `"helm.sh/hook-delete-policy": before-hook-creation,hook-succeeded`. This immediately cleans up successful Jobs rather than relying on TTL. Failed Jobs are intentionally retained for diagnosis (current behavior preserved).

### 5. Per-service overlay `USER root` mid-build then `USER migrator` not enforced

- **Where:** `ops/migrations/README.md:66-79` (canonical overlay Dockerfile example)
- **What:** The example overlay runs `USER root` before `RUN bun install --frozen-lockfile --production` (line 71), then `USER migrator` at the end (line 79). This is a pattern risk: if a service author omits the final `USER migrator` line, the container runs as root despite the pod-level `runAsNonRoot: true` securityContext — which Kubernetes enforces only at container startup, not at build time. The base image correctly ends with `USER migrator` (Dockerfile line 41), but the overlay explicitly breaks that with `USER root`.
- **Fix:** In the example overlay, add a Dockerfile linter step (Hadolint rule DL3002: avoid `USER root`) to the service CI check, or restructure the overlay to use `--mount=type=cache,uid=1000` or `RUN --mount` for bun install without dropping to root.

### 6. `MIGRATION_STATEMENT_TIMEOUT_MS` default 60 s — inadequate for large-table DDL

- **Where:** `ops/migrations/run-migrations.ts:29`; `ops/migrations/job-template.yaml:107`
- **What:** `MIGRATION_STATEMENT_TIMEOUT_MS` defaults to 60,000 ms (1 minute). The `activeDeadlineSeconds: 900` on the Job (15 min) implies the PR authors expect migrations could take up to 15 minutes. But if any single DDL statement (e.g., `CREATE INDEX` on a 50M-row table) exceeds 60 s, Postgres cancels it with `ERROR: canceling statement due to statement timeout`, the migration fails, the Job retries via `restartPolicy: OnFailure`, and the index is not created but the retry re-runs the same `CREATE INDEX` (idempotent only if `IF NOT EXISTS` — see P0-3 above). The statement timeout is configurable via env, but the Helm template hardcodes `60000` with no comment or operator guidance on when to raise it.
- **Fix:** Either raise the default to match `activeDeadlineSeconds` (e.g., 840,000 ms = 14 min), or document explicitly in `job-template.yaml` that `MIGRATION_STATEMENT_TIMEOUT_MS` must be set per-service for any migration touching large tables. Add a `TODO(migration-author)` inline comment.

### 7. PITR runbook missing app-pod stale schema cache recovery step

- **Where:** `ops/migrations/forward-only-policy.md:30-56` (PITR runbook)
- **What:** The runbook scales the service to 0 replicas (step 1), restores a new CNPG cluster (steps 3-4), then scales back up (step 6). It does not address: app pods that were running between step 1 and step 6 may have cached the post-migration schema in Drizzle's ORM schema introspection cache or in pgBouncer's prepared-statement cache. After the CNPG cluster is restored to the pre-migration point, any pgBouncer instance that has cached the post-migration table definitions will serve stale metadata until it is restarted. For CNPG with `pgbouncer.poolMode: transaction` (default per `curaos_postgres_rule.md:184`), the risk is lower but not zero for `SET` commands cached in pgBouncer session state.
- **Fix:** Add step between 4 and 5 in the PITR runbook: `kubectl rollout restart deploy/pgbouncer -n <ns>` to flush pooler connection state, and a note that all service Deployments must be restarted (not just scaled) to invalidate any in-process schema cache.

### 8. Forward-only CI gate claims `drizzle-kit check` blocks destructive statements — false

- **Where:** `ops/migrations/forward-only-policy.md:25`
- **What:** "`drizzle-kit check` ... block PRs containing destructive statements without an `-- forward-only-policy: justified` comment block." `drizzle-kit check` does NOT lint SQL statement bodies. It validates that the schema source matches the migration snapshot. It does not parse individual SQL files for `DROP`, `ALTER COLUMN TYPE`, or absence of `IF NOT EXISTS`. The claim that `drizzle-kit check` blocks destructive statements is factually wrong.
- **Fix:** Remove the `drizzle-kit check` claim from item 5 of `forward-only-policy.md`. The enforcement must come entirely from `scripts/check-migrations.ts` (which does not yet exist — see P0-3). Until that script ships, the correct text is "no automated CI gate currently enforces statement-level safety; each migration PR requires human review."

### 10. `console.error('[migrator] FAILED', err)` — may log DB error messages containing query fragments

- **Where:** `ops/migrations/run-migrations.ts:58`
- **What:** `console.error('[migrator] FAILED', err)` logs the full `Error` object. Postgres `Error` objects in `pg` (node-postgres) include `message`, `detail`, `hint`, and `query` fields. If a migration fails mid-INSERT (e.g., a seed/backfill migration), the `query` field may contain literal SQL with data values. For health data migrations, this could include PHI embedded in `WHERE` or `VALUES` clauses.
- **Fix:** Replace `console.error('[migrator] FAILED', err)` with a sanitized log: `console.error('[migrator] FAILED', { message: err.message, code: (err as any).code, severity: (err as any).severity })`. Never log `err.query` or `err.detail`.

### 11. No RBAC for `{{ .Values.service.name }}-migrator` ServiceAccount

- **Where:** `ops/migrations/job-template.yaml:63` — `serviceAccountName: "{{ .Values.service.name }}-migrator"` is referenced but not created by this PR.
- **What:** The ServiceAccount must be created before the Job runs. If the service Helm chart does not ship a `ServiceAccount` + `Role` + `RoleBinding` for the migrator SA, the Job pod fails to start. The PR assumes the per-service chart creates this SA but provides no template or validation.
- **Fix:** Add a note in `README.md` under "Helm values wiring" that the service chart must include a `ServiceAccount` YAML (or reference a shared chart helper) with `automountServiceAccountToken: false`.

## What Claude got right

1. **Pod-level security hardening is solid.** `securityContext.runAsNonRoot: true`, `runAsUser: 1000`, `readOnlyRootFilesystem: true`, `capabilities.drop: ["ALL"]`, `allowPrivilegeEscalation: false`, and `seccompProfile.type: RuntimeDefault` are all present on both the init-container (lines 89-93) and main container (lines 117-121). This meets the container security baseline without needing a PSA policy.

2. **Air-gap runtime isolation is correct.** `imagePullPolicy: IfNotPresent` on all containers, no `bun install` at runtime in `run-migrations.ts`, `DATABASE_URL` injected via `secretKeyRef` from a CNPG-generated secret, no outbound DNS calls. The design correctly avoids registry dependency at deploy time.

3. **`activeDeadlineSeconds: 900` hard deadline prevents indefinite blocking.** The Job has a 15-minute wall-clock ceiling, which prevents a wedged migration from permanently blocking `helm upgrade`. Combined with `backoffLimit: 3`, the failure surface is bounded.

4. **`tini` as PID 1 for clean signal propagation.** `ENTRYPOINT ["/sbin/tini", "--", "bun", "run", "run-migrations.ts"]` ensures SIGTERM from a preempted node propagates correctly to the Bun process, enabling graceful pool shutdown via the `catch` block in `run-migrations.ts:59`.

5. **Expand-and-contract pattern documented clearly.** The 3-release lifecycle in `forward-only-policy.md:22-26` (add → stop reading → drop) is the correct zero-downtime DDL pattern and is explicitly required. This alone prevents the most common migration data-loss failure mode in continuous-delivery systems.

6. **`hook-delete-policy: before-hook-creation` prevents duplicate Jobs on rapid re-upgrade.** The existing annotation ensures the old Job is torn down before a new one is created for the same Helm hook event, preventing the simplest form of concurrent execution (though not the cross-revision form identified in P0-2).
