## Findings

### PR333-P1-1
Severity: P1
file: tools/codegen/templates/service-core/chart/templates/secret.yaml:35
BREAK: The PHI invariant is not enforced. The template says PHI DSNs are "never a chart-managed Secret" at `tools/codegen/templates/service-core/chart/templates/secret.yaml:22-24`, but the implementation blindly copies every `.Values.secret.data` key/value into `stringData`: `tools/codegen/templates/service-core/chart/templates/secret.yaml:35` is `{{- $data := .Values.secret.data | default dict }}` and `:37` is `{{ $key }}: {{ $value | quote }}`. A values override such as `secret.data.SUBJECT_RIGHTS_CERTIFICATE_DATABASE_URL=postgresql://...` renders that PHI DSN into the chart-managed Secret manifest, then Kubernetes stores it under Secret `data` and `envFrom.secretRef` injects it via `tools/codegen/templates/service-core/chart/templates/deployment.yaml:47-51`. This violates the stated requirement in `tools/codegen/__tests__/templates/chart-secret-env-684.test.ts:26`, `PHI DSNs are NEVER stored in the chart Secret (no secret.data DSN key)`.
FIX: Make the invariant executable. Either split JWT material into a dedicated `secret.jwt` map with only the two JWT keys, or add Helm `fail` plus values schema validation that rejects `DATABASE_URL`, `*_DATABASE_URL`, `SUBJECT_RIGHTS_CERTIFICATE_DATABASE_URL`, and `BREAK_GLASS_AUDIT_EVIDENCE_DATABASE_URL` under `.Values.secret.data`. Require PHI DSNs to use `secretEnv` only.

### PR333-P1-2
Severity: P1
file: tools/codegen/templates/service-core/chart/values.yaml.hbs:62
BREAK: PHI DSN fail-closed is opt-in, not guaranteed. The design says every emitted service chart must prevent start without "durable PHI evidence DSNs" at `tools/codegen/__tests__/templates/chart-secret-env-684.test.ts:4-6`, but the default chart has `secretEnv: []` at `tools/codegen/templates/service-core/chart/values.yaml.hbs:62`. The Deployment only emits PHI `secretKeyRef` entries inside `{{- with .Values.secretEnv }}` at `tools/codegen/templates/service-core/chart/templates/deployment.yaml:52`, so default values produce no `SUBJECT_RIGHTS_CERTIFICATE_DATABASE_URL` or `BREAK_GLASS_AUDIT_EVIDENCE_DATABASE_URL` env var, and Kubernetes has no missing Secret/key to fail on. The repo search found those two PHI DSN names only in this new chart test/template surface, not in a service boot validator. Concrete hole: a service with valid JWT env and default `secretEnv: []` can start without either PHI DSN from this chart layer.
FIX: For services that require the PHI evidence DSNs, generate required non-optional `secretEnv` entries by default or via a required per-service list, and add service boot validation that throws when a required PHI DSN env is missing or empty. If only some services require them, gate the required list by service or overlay, but do not leave the required DSNs as comments.

### PR333-P2-1
Severity: P2
file: tools/codegen/templates/service-core/chart/templates/deployment.yaml:66
BREAK: Required `secretEnv` entries can be made optional by values. The template renders `optional` whenever the entry contains the key: `tools/codegen/templates/service-core/chart/templates/deployment.yaml:66-67` is `{{- if hasKey . "optional" }}` and `optional: {{ .optional }}`. For the PHI DSN names, setting `optional: true` disables the missing Secret/key failure that lines `:56-58` claim for required keys.
FIX: Reject `optional: true` for required env names, especially `DATABASE_URL`, `SUBJECT_RIGHTS_CERTIFICATE_DATABASE_URL`, and `BREAK_GLASS_AUDIT_EVIDENCE_DATABASE_URL`, with Helm `fail` or schema validation.

### PR333-P3-1
Severity: P3
file: /tmp/pr333.diff:180
BREAK: The diff itself adds prohibited dash characters in the test regex. To avoid re-emitting them here, the line is represented with escaped code points: `expect(src).not.toMatch(/[\u2014\u2013]/);` (the literal characters are escaped here as code points). This violates the "em/en dash anywhere in the diff" hygiene gate even though the test is trying to prevent those characters in chart files.
FIX: Write the regex with escapes in source, for example `expect(src).not.toMatch(/[\\u2014\\u2013]/u);`, so the diff contains no literal U+2014 or U+2013 characters.
