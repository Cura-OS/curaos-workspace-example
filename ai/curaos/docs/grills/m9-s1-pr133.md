## Verdict

**APPROVE WITH CONDITIONS** — the Diamond model architecture is the correct choice and the core reasoning holds, but two structural defects must be resolved in the ADR before M9-S2 starts: (1) cross-service Postgres FKs between separately-owned tables violate microservice data isolation and will cause distributed transaction failures; (2) `identities.email` has a global UNIQUE constraint that breaks multi-tenant operation at Citus scale. Neither is a "prose" issue — both will force schema redesign mid-stream if uncorrected.

## P0 — Architectural Defects (force M9-S2/3/4 redesign if unresolved)

### P0.1 — Cross-service Postgres FKs violate microservice data isolation

**Finding:** The ADR assigns table ownership across three separate services: `identity-service` owns `actors` + `identities`; `party-core-service` owns `parties`; `org-core-service` owns `orgs` + `actor_memberships`. Yet the DDL mandates Postgres FKs across these service boundaries:

```sql
-- parties.actor_id → actors.id   (party-core-service → identity-service's table)
-- orgs.actor_id → actors.id      (org-core-service → identity-service's table)
-- actor_memberships.actor_id → actors.id   (org-core-service → identity-service's table)
-- actor_memberships.org_id → orgs.id       (cross-table within org-core-service — OK)
```

**Evidence:**
- ADR §"Service ownership": `party-core-service` owns `parties`; `org-core-service` owns `orgs` + `actor_memberships` (ADR-0210 lines 52-54).
- Spike §3.1 DDL: `parties.actor_id UUID NOT NULL UNIQUE REFERENCES actors(id) ON DELETE RESTRICT` — hard Postgres FK from party-core-service's table into identity-service's `actors` table (spike lines 81-88).
- Same pattern: `orgs.actor_id ... REFERENCES actors(id)` and `actor_memberships.actor_id ... REFERENCES actors(id)` (spike lines 95-113, 119-129).

**Why this is a P0:** In the CuraOS model, each service owns its own DB schema. Postgres FK enforcement requires both tables to be in the same DB connection scope. If `parties` is in `party-core-service`'s schema and `actors` is in `identity-service`'s schema, the FK cannot be enforced at the DB level — Postgres does not enforce FKs across schemas in different transactions from different services. The FK becomes a documentation-only constraint, silently unenforced. Every M9-S3/S4 scaffold will be built assuming FK integrity that does not exist. Cascades (`ON DELETE RESTRICT`) will not fire.

**Required fix before M9-S2:**

Option A (preferred): All five Diamond tables (`actors`, `parties`, `orgs`, `identities`, `actor_memberships`) reside in a **single shared DB schema** owned by `identity-service`. Other services (`party-core-service`, `org-core-service`) become **domain modules within identity-service's NestJS monorepo module boundary** rather than separately deployed services with separate DBs. Cross-service writes happen at the module layer, not via Kafka choreography. This is consistent with the `curaos_modulith_standalone_rule.md` (modulith first, extract when justified).

Option B: Cross-service integrity is enforced at the **application layer** only (no DB-level FKs between service schemas). ADR must explicitly document that `parties.actor_id` is a **soft reference** (UUID stored, no DB FK), integrity enforced by event-driven choreography, and the `ON DELETE RESTRICT` in the DDL sketch applies only if deployed as a single-schema modulith. The DDL sketch in the spike must be annotated accordingly — as-is it is misleading.

The ADR is silent on which DB topology these tables share. This silence is the defect.

### P0.2 — `identities.email` global UNIQUE constraint breaks multi-tenant Citus sharding

**Finding:** The spike DDL declares `UNIQUE (email)` on `identities.email` without a `tenant_id` scope:

```sql
CREATE TABLE identities (
  ...
  email TEXT NOT NULL,
  ...
  UNIQUE (email)                -- ← global, not per-tenant
);
```

**Evidence:** Spike §3.1, identities table DDL (spike lines ~103-115). The `UNIQUE (email)` constraint is on `email` alone.

**Why this is a P0:** Per `curaos_postgres_rule.md`, the tenant isolation model for neutral high-volume services is **Citus shared schema sharded by `tenant_id`**. In this topology, the same physical DB holds rows from all tenants distinguished by `tenant_id`. A global `UNIQUE(email)` means user `doctor@hospital-a.com` in tenant A cannot also exist as `doctor@hospital-a.com` in tenant B — even though these are entirely separate tenants with no relationship. Email addresses are tenant-scoped identifiers. The correct constraint is `UNIQUE(tenant_id, email)`.

Additionally, the Drizzle schema sketch (spike §3.3) uses `unique('identities_email_unique').on(t.email)` — same defect in the code path that M9-S2 will copy into generated schema files.

**Required fix:** Change to `UNIQUE(tenant_id, email)` in both the SQL DDL and Drizzle sketch. Add `tenant_id` column to `identities` table (or confirm that `actors.tenant_id` is accessible via JOIN and enforce the constraint there via a partial index). Simplest fix: add `tenant_id UUID NOT NULL REFERENCES tenants(id)` to `identities` (denormalized for query performance) + `UNIQUE(tenant_id, email)`.

## P1 — Missing Considerations (must fold into ADR before M9-S2 starts)

### P1.1 — Orphan `actors` rows: no DB-level enforcement of minimum peer

**Finding:** The ADR states `actor_type` tells consumers "which peer tables a given actor row participates in," but there is no DB-level constraint ensuring that an `actor_type='human'` row actually has a corresponding `parties` row, or that an `actor_type='org'` row has a corresponding `orgs` row. The FK direction is inward (peers → actors), so `actors` cannot know whether peers exist.

**Evidence:** ADR §Decision: "The `actor_type` column tags which peer tables a given actor row participates in" — no enforcement mechanism named. Spike §3.1: no trigger, deferred constraint, or CHECK referencing peer tables visible in the DDL. ADR §Consequences (Negative): silent on orphan risk.

**Why P1 (not P0):** This is a data integrity gap, not a fundamental model error. The Diamond model is still correct. But if identity-service crashes between writing `actors` and emitting `ActorCreated` (before party-core-service creates the `parties` row), a permanent orphan `actor_type='human'` row with no `parties` row exists. Any "get practitioner demographics" query silently returns null.

**Recommended addition:** ADR should specify the consistency mechanism: (a) transactional write if same-schema modulith (P0.1 Option A resolves this automatically), OR (b) outbox + retry + dead-letter reconciliation job + alerting on `actors` rows older than N minutes with no matching peer row for their `actor_type`. The spike mentions outbox choreography but does not address the inconsistent-state detection gap.

### P1.2 — `actor_memberships` UNIQUE constraint allows role change but not detection

**Finding:** `UNIQUE(actor_id, org_id, membership_type)` means a practitioner can hold at most one row per `(actor, org, type)`. But `role` is a separate column — a practitioner at org X can have their role changed by UPDATEing the `role` column. This is not audited at the DB level, and the ADR does not specify whether role changes go through the outbox.

**Evidence:** Spike §3.1 `actor_memberships` DDL: `role TEXT NOT NULL` with no FK to a roles table and no trigger/outbox reference. ADR §Consequences: references RBAC service interprets `role` but does not state that writes to `role` emit audit events.

**Recommended addition:** ADR §Implementation Plan should explicitly state that UPDATE on `actor_memberships.role` must go through identity-service's outbox (emit `MembershipRoleChanged` event) for audit chain integrity. If M9-S4 (org-core-service) owns this table, the outbox responsibility must be in org-core-service's AGENTS.md, not identity-service's. This is a contract gap that will be debated mid-M9-S4 without a clear ruling.

### P1.3 — FHIR Patient is not an actor — no mapping specified

**Finding:** The ADR maps FHIR `PractitionerRole` → `actor_memberships` (correct). But FHIR `Patient` is a distinct resource that does NOT map to `Practitioner`. In HealthStack overlays, a Patient is a person (`parties` row) but is NOT a practitioner and does NOT participate in `actor_memberships`. The ADR does not specify how HealthStack's `Patient` resource attaches to the Diamond — whether a Patient gets an `actors` row at all, what `actor_type` it would carry, and whether Patient PHI flows through `parties.date_of_birth` (which the spike includes in the schema).

**Evidence:** Research §D2 line 106: "FHIR uses a flat resource model. `Patient` is a resource; `Organization` is a resource; `Practitioner` is a resource." — the research names all three but the ADR's FHIR mapping section discusses only `PractitionerRole`. Spike §3.2 bullet 5 names cross-tenant practitioners, not patients. ADR §Consequences (Positive): "No dummy / synthetic rows" — this claim assumes Patient is handled, but no mapping is given.

**Recommended addition:** One paragraph in ADR §Consequences or a new §FHIR overlay mapping: "HealthStack `Patient` resource maps to a `parties` row (PHI) with a corresponding `actors` row (`actor_type='human'`). Patient does NOT participate in `actor_memberships` (patients are not members of orgs in the FHIR sense; clinical encounters link patient to org via `Encounter.serviceProvider`). The HealthStack patient overlay attaches via FK to `party_id` per the PHI boundary rule." Without this, M9-S3 will make an undocumented decision about whether patients get `actor_memberships` rows.

### P1.4 — UUID v4 in DDL, UUID v7 promised — discrepancy in generated code

**Finding:** The SQL DDL in spike §3.1 uses `DEFAULT gen_random_uuid()` (UUID v4). The Drizzle sketch uses `.defaultRandom()` (also v4). The spike header and comments say "emit UUID v7 in app layer" and "UUID v7 for monotonic ordering" in research §D3. The ADR §Consequences (Neutral) is silent on which version lands in which layer.

**Evidence:** Spike §3.1 DDL comment: `DEFAULT gen_random_uuid(),   -- emit UUID v7 in app layer`. Drizzle sketch: `uuid('id').primaryKey().defaultRandom()` — `defaultRandom()` in Drizzle calls `gen_random_uuid()` at the DB level (v4). Research §D3 audit schema: `eventId: z.string().uuid()  // UUID v7 for monotonic ordering`.

**Why this matters for M9-S2:** If M9-S2 copies the Drizzle sketch directly, `actors.id` will be generated as UUID v4 at the DB level. The M9 audit chain uses UUID v7 for event ordering. An `actors.id` that is v4 is referenced as `actor_id` in audit events — the audit chain's ordering guarantee relies on the event's own UUID v7, not on `actor_id`, so this is not catastrophic. But the intent mismatch will cause confusion in M9-S2 code review. The ADR should state explicitly: "UUID v7 is generated in the application layer before INSERT; the DB DEFAULT is v4 fallback only; codegen templates must call the UUID v7 utility."

### P1.5 — Soft-delete deferred but downstream impact on audit chain not specified

**Finding:** ADR §Open Questions defers "Soft-delete vs hard-delete on `actor_memberships`" to M9-S4. But the ADR does not defer the same question for `actors`, `parties`, `orgs`, or `identities`. All four peer tables use `ON DELETE RESTRICT` in the DDL — hard-delete is blocked but no soft-delete mechanism is defined for these tables.

**Evidence:** Spike §3.1: `ON DELETE RESTRICT` on all peer tables' `actor_id` FKs. ADR §Open Questions: only `actor_memberships` deletion deferred. ADR §Consequences: silent on deletion behavior for `actors` / `parties` / `orgs` / `identities`.

**Why this matters before M9-S2:** If identity-service (M9-S2) generates the `actors` + `identities` schema without a `deleted_at` column, adding soft-delete later requires a schema migration + backfill across all existing rows. GDPR subject erasure (referenced in `curaos_postgres_rule.md` §GDPR erasure) requires either hard-delete OR a documented pseudonymization approach. The ADR should specify: soft-delete columns (`deleted_at TIMESTAMPTZ`) on at minimum `actors` and `identities`, with a hard-delete path for GDPR subject erasure requests.

## P2 — Clarity and Polish

- **Star vs Diamond naming:** The ADR introduces "Diamond model" but never defines it vs "Star schema" in the text. A reader unfamiliar with the research doc has no anchor. One sentence defining Diamond (four peer tables, no root, inward FKs to shared UUID) would prevent misreading.
- **`actor_memberships.role` is a free-text string:** No FK to a roles table, no CHECK constraint. The ADR says "RBAC service interprets" — this is fine architecturally but the ADR should note that `role` values are RBAC-service-defined and the schema carries no local validation. Otherwise M9-S4 developers will add a CHECK and conflict with the RBAC service's role vocabulary.
- **Materialized view `actor_primary_org` freshness SLA is deferred to M14:** This is a long deferral. The view is proposed as the primary performance mitigation for the Diamond JOIN cost, yet the performance SLO (P95 < 250ms) is an M9-S8 target. The ADR should at minimum specify the staleness tolerance at M9 (e.g., "acceptable up to 5s stale during M9; tightened at M14") rather than leaving the view undefined for 5+ milestones.
- **`external_subject` UNIQUE scope deferred to M9-S5:** Reasonable deferral, but the spike §6 risk table says `UNIQUE(external_subject, issuer)` is the fix. The DDL currently has no `issuer` column. M9-S2 should not add a `UNIQUE(external_subject)` constraint that later M9-S5 must drop and replace — the ADR should instruct M9-S2 to leave `external_subject` without a uniqueness constraint until M9-S5 adds it with the `issuer` column.
- **Backfill `display_name → given_name + family_name` split:** The spike flags "best-effort split + flag `parties.metadata.split_source = 'm3_backfill'`" but `parties` has no `metadata` column in the DDL. Either add `metadata JSONB` (and document PHI scrub treatment) or change the flag to a dedicated boolean column `name_split_from_display_name BOOLEAN DEFAULT false`.
- **`orgs_org_type_check` CHECK constraint:** `external` as an org type is ambiguous — external to the tenant, to the facility, or to the CuraOS deployment? The value should be renamed or documented. This is a naming issue, not an architecture issue.
- **Spike §5.2 cutover Phase 2 materializes `user_roles → actor_memberships` with `membership_type='staff'`:** All M3 users become `staff` members. Clinician users who were `practitioners` in M3 business logic are silently downgraded. ADR should note this and specify that a follow-up migration (M9-S10 or M11) re-classifies based on M3 role name patterns.

## What Claude Got Right

- **Diamond model selection is correct.** The five-case analysis (cross-tenant practitioners, HIPAA BAA, system actors, service accounts, audit subject uniformity) is thorough and the model wins on every axis. Nested-A (Party root) and Nested-B (Org root) rejections are well-reasoned and match the FHIR / SCIM prior art.
- **`actor_memberships` unique constraint is well-formed.** `UNIQUE(actor_id, org_id, membership_type)` correctly allows a practitioner to hold multiple roles at the same org (different `membership_type`) while preventing duplicate membership rows. Time-windowing via `valid_from` / `valid_until` is the right pattern for temporal membership.
- **Credentials out of `identities`.** Keeping credential bytes in a separate auth-service domain is the correct PHI / security boundary. `identities` holds only the auth subject reference (`email`, `external_subject`), not password hashes. This matches the Cognito User Pool / Identity Pool split prior art.
- **`actors.tenant_id` as defense-in-depth.** Adding `tenant_id` on the root `actors` table (in addition to per-tenant schema isolation) is the correct defense-in-depth guard against cross-tenant JOIN leaks in Citus queries.
- **Strangler Fig dual-write with dedup by `event_id` UUID v7.** The migration sequencing is competent. Using the same `actors.id` as M3's `users.id` preserves JWT `sub` claim validity through cutover — this is the correct approach and avoids a token invalidation event.
- **HIPAA BAA modeled as `membership_type='business_associate'`.** This is the cleanest representation; no special BAA table needed. The Covered Entity / Business Associate relationship is structurally identical to a practitioner-at-org relationship, and the ADR correctly exploits that symmetry.
- **No jsonb for party PHI fields.** `given_name`, `family_name`, `date_of_birth` are typed columns (`TEXT`, `DATE`), not JSONB blobs. PHI scrub via Presidio (per `curaos_agent_eval_obs_rule.md`) can target these columns by name. This is correct.
- **ORM alignment with `curaos_orm_rule.md`.** Drizzle for all five tables is correct; none are on the MikroORM clinical-aggregate allowlist. The Drizzle sketch is idiomatic and matches current Drizzle v0.30+ API patterns.
- **Materialized view `actor_primary_org` as JOIN cost mitigation.** Acknowledging the Diamond JOIN cost and specifying a materialized view + index on `(actor_id, membership_type)` is the right performance mitigation. The P95 < 250ms SLO (M9-S8 #105 per research §D2) is achievable with this mitigation in place.
- **`parent_org_id` self-referential on `orgs` with `ON DELETE RESTRICT`.** Correct for org hierarchy. Blocking delete of parent orgs before children prevents orphaned org subtrees without requiring cascade behavior.
