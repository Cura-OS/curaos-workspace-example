## 1. Missing Questions

**MQ-1: `valueset-updated` trigger semantics undefined.**
Issue says "publish via durable outbox" but never defines WHEN the event fires. Options: (a) on every $expand call, (b) only when a tenant ValueSet is mutated/created in HAPI JPA, (c) only when a ConceptMap translation cache is refreshed. These are wildly different frequencies and semantics. The acceptance test only asserts the topic string and empty consumer set — not trigger conditions. Without this, the implementer will guess.

**MQ-2: `suggest` response shape unspecified.**
Issue says `GET /terminology/suggest?text=&system=snomed` per ADR-0208 §3.15, but ADR-0208 doesn't exist in the repo (`ai/` tree has no file matching `0208`). No acceptance test covers `/suggest`. No response schema (list of concepts? FHIR Parameters? flat JSON?). Implementer must invent the contract.

**MQ-3: Air-gap "bundled RF2/Lite" scope for this issue.**
Issue says "Air-gap RF2 bundles per national edition via Zarf" but that's packaging infra — not a TypeScript task. The acceptance test only asserts `SNOWSTORM_URL` is undefined; it does NOT test that a bundled fallback actually returns any concept. Plan says "tiny deterministic sample concepts" — is that the full air-gap story here, or does actual RF2 bundle loading belong to a separate Zarf story?

**MQ-4: Jurisdiction check — module activation vs. query-time.**
Acceptance test locks "enforced at module activation, not at query time." The plan says nothing about NestJS module lifecycle hooks (`OnModuleInit` / `OnApplicationBootstrap`). Is `snomedActivationAllowed()` a pure function called during bootstrap that throws if config is invalid, or a guard on the `TerminologiesModule.register()` options? These require different wiring.

**MQ-5: ConceptMaps (ICD-10→SNOMED, LOINC→SNOMED) — bundled or live-only?**
Issue lists "ICD-10-CM→SNOMED (NLM GEM) + LOINC→SNOMED Observable-Entity ConceptMaps; tenant value sets in HAPI JPA, versioned in Apicurio." In air-gap mode with no HAPI URL, does `$translate` return 501/empty, or is a bundled NLM GEM ConceptMap expected? This affects the fallback layer complexity significantly.

## 2. Docs/ADR Conflicts

**DC-1: AsyncAPI spec topic namespace conflict.**
The existing `specs/terminology.asyncapi.yaml` uses `curaos.core.terminology.{created,updated,deleted}.v1` — a CRUD model. The issue requires adding `healthstack.terminology.valueset-updated` — a **different namespace** (`healthstack.*` vs `curaos.core.terminology.*`). The spec header says "Topics use the `curaos.core.terminology.*` namespace per the shared event-producer template." The `healthstack.*` topic breaks this namespace contract. The AsyncAPI spec will need a new channel that crosses the established namespace boundary — this isn't just adding a channel, it's a namespace exception that needs explicit rationale.

**DC-2: `domain-event-catalog.ts` scaffold says "Replace/extend" — but it's generated.**
The file carries `Generated by @curaos/codegen — see [[curaos-generator-evolution-rule]]`. The plan says to replace `TerminologyRecorded` / `curaos.core.terminology.recorded.v1` with the `valueset-updated` event. Doing so is a **generator-evolution** operation, not a local edit — the generator mold must be updated or the divergence classified. The issue says "Do not bypass generator-evolution gates" but gives no generator ticket or explicit evolution classification.

**DC-3: `amount_minor` / dual-emit in `domain-event-catalog.ts` is dead weight here.**
The scaffold's `TerminologyRecorded` payload has `amount_minor` + `amount_minor_str` money fields. FHIR terminology operations carry no money. The `valueset-updated` event will have none either. These money helpers will still be in the file (or need deletion). Either way the plan must explicitly decide: keep helpers as dead code (violates "delete unused"), or delete them (requires understanding what `buildTerminologyRecordedPayload` callers exist — none currently, scaffold only).

**DC-4: Route prefix `/terminologies` vs FHIR convention `/fhir`.**
Controller is `@Controller('terminologies')`. Plan adds `GET /fhir/ValueSet/$expand` etc. under a new `@Controller('fhir')` or adds new routes to the existing controller? If a second controller: module must register it. If added to existing controller with `@Get('fhir/ValueSet/$expand')` — the `$` character in NestJS route paths is legal but untested in this codebase and may need URL encoding. The plan doesn't specify which controller gets the FHIR routes.

## 3. Glossary Conflicts

**GC-1: "air-gap mode" overloaded.**
Issue uses "air-gap" to mean "no external network call." Plan uses "in-memory bundled reference fallback with tiny deterministic sample concepts." These are not the same: the issue's AC #3 mandates zero external calls (a network policy), not a specific data source. A valid air-gap can return 404/empty for unknown codes — it doesn't require bundled sample data. The plan conflates "no network" with "must return results."

**GC-2: "reference-only" boundary.**
CONTEXT.md says "no PHI" and "reference-only." The plan says `recordValueSetUpdated()` with "reference-only fields." But `valueset-updated` isn't defined anywhere — neither the payload schema nor which fields qualify as reference-only vs PHI-adjacent (e.g., is `tenant_id` reference-only? Is a `valueSetUrl` pointing to a tenant-scoped value set reference-only?). The verification step says "Run PHI/reference-only checks" but there's no defined checklist.

## 4. Hidden Deps / Subtasks

**HD-1: `$` in NestJS route path.**
`@Get('fhir/ValueSet/$expand')` — NestJS uses Express/Fastify routing where `$` has no special meaning, but must confirm the FHIR operation name routes correctly and isn't stripped/rejected by any middleware. Needs a route smoke-test, not assumed.

**HD-2: `DomainOutboxEnqueueInput.messageKey` must be set.**
`enqueue()` requires `{ tenantId, topic, messageKey, value, headers, idempotencyKey }`. The `messageKey` is `partitionKey(tenantId, resourceId)` per the catalog's `buildDomainMessage`. The `valueset-updated` event needs a stable `resourceId` to partition on — what is the aggregate ID for a ValueSet update? The plan says nothing about this.

**HD-3: `TerminologiesService` has no `DomainOutboxService` injection yet.**
`terminologies.service.ts` only has `status()`. Adding outbox publishing requires injecting `DomainOutboxService` — the module already imports `DomainOutboxModule.register()`, so the provider is available, but the service constructor must be updated.

**HD-4: SNOMED jurisdiction check wiring location.**
Acceptance harness tests `snomedActivationAllowed()` as a pure function. But the issue says "enforced at module activation." The implementer must export this function from the service/module AND call it during bootstrap — two separate tasks. The plan only mentions it as a function call.

**HD-5: AsyncAPI `@asyncapi/parser` validation gate.**
The spec file header says "Validate: `@asyncapi/parser`." `bun run ci` must include this. Check if it's in the current `ci` script — if not, adding a new AsyncAPI channel without adding the parser to CI means the contract can silently break.

**HD-6: `suggest` endpoint has no test.**
Acceptance test file has zero coverage of `/suggest`. The plan adds the route, but there is no test to lock the contract. With TDD required, a test must precede the implementation.

## 6. Decision Points — Implementer Must Auto-Apply

**DP-1: Route placement for FHIR operations.**
**Decision:** Add a second `FhirTerminologyController` with `@Controller('fhir')` registered in `TerminologiesModule`. Do NOT add `$`-suffixed routes to the existing `@Controller('terminologies')` controller — this keeps the auth-matrix locked routes untouched per [[curaos-generator-evolution-rule]]. Apply `@UseGuards(AuthGuard, RolesGuard)` + `@RequiresTerminologyScope({ action: 'read', roles: Terminology_READ_ROLES })` class-level — same pattern as existing controller.

**DP-2: `suggest` route placement.**
**Decision:** `GET /terminology/suggest` → add to existing `TerminologiesController` (not a new controller) since it's under the `terminologies` prefix and is a NestJS read endpoint, not a FHIR protocol operation. Apply `@RequiresTerminologyScope({ action: 'read', roles: Terminology_READ_ROLES })`.

**DP-3: Keep vs. delete scaffold money helpers in `domain-event-catalog.ts`.**
**Decision:** Delete `buildTerminologyRecordedPayload`, `SafeMinor`, `amount_minor` helpers when replacing the scaffold event — they have zero callers outside the scaffold example. Keeping dead money code in a service with no financial domain is a mislead. Classify as scaffold replacement per generator-evolution (in-service evolution, no new generator ticket needed since the scaffold explicitly says "Replace with the service's real channel set").

**DP-4: `valueset-updated` event `messageKey` / partition subject.**
**Decision:** Use `valueSetUrl` (the FHIR ValueSet canonical URL) as the aggregate ID for `partitionKey(tenantId, valueSetUrl)`. This gives per-ValueSet ordering which matches the event semantics. Never use a request-derived actor ID.

**DP-5: Air-gap fallback scope.**
**Decision:** Air-gap mode = return structured empty/404 FHIR OperationOutcome for unknown codes — do NOT bundle real RF2 data in TypeScript. The acceptance test only asserts network silence, not data presence. Bundled RF2 is a Zarf/infra story (out of scope per issue). The "tiny deterministic sample" in the plan is fine for unit test fixtures only — not the production fallback path.

**DP-6: `TerminologyRecorded` scaffold event — retain or remove.**
**Decision:** Remove `Terminology_RECORDED_TOPIC` / `TerminologyRecorded` type / `buildTerminologyRecordedPayload`. Replace with `VALUESET_UPDATED_TOPIC = 'healthstack.terminology.valueset-updated'` + typed `ValueSetUpdatedPayload`. This is the explicit issue requirement ("Replace the generic … example topic").

**DP-7: AsyncAPI namespace exception.**
**Decision:** Add `healthstack.terminology.valueset-updated` as a new channel with an explicit comment documenting the namespace exception (healthstack vs curaos.core). Do NOT rename existing channels. The comment is the evolution record; no separate ADR needed for a namespace prefix on a new topic.

**DP-8: `suggest` response shape (pending MQ-2 resolution, but implementable with a safe default).**
**Decision:** Until ADR-0208 is located/created, return a typed array of `{ code: string; display: string; system: string }` — the minimal FHIR Coding shape. This is the least-surprising contract for a coding-assist endpoint and doesn't force consumers into FHIR Parameters parsing.
