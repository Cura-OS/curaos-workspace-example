## Why this exists

M6 introduces the first "Mold" - a repeatable scaffold that emits a CuraOS-compliant submodule (NestJS service or shared package) with all the boilerplate the modulith doctrine + ASDLC discipline demands. The prior `tools/generators/` package (a single `@clack/prompts` CLI) covered M0-M5 prototype work but cannot express the M6 surface:

- 3 service layers (`<name>-core-service`, `personal-<name>-service`, `business-<name>-service`) - the default trio. Two opt-in single-root variants share the `service-core` tree: `--core-only` → `<name>-core-service` (ADR-0210 party/org/audit) and `--plain-service` → plain `<name>-service` (ADR-0201 §1 settings/notify/storage/search/reports, #290). The `plain` layer is never in the default set.
- Auto-wiring (AppModule registration via `ts-morph`).
- Dual-mode doc emission (opinionated when `--domain` + `--purpose` provided; stub otherwise).
- Workspace integration (lefthook, .npmrc, Verdaccio, DOC-GRAPH, ai/curaos mirror).

The hybrid stack distributes these concerns across three best-of-breed tools rather than re-implementing them in one script.

### Why each layer is non-negotiable

- **Nx** brings mature playbook composition (prompts, schemas, hooks) without taking over the build. By restricting Nx to `nx generate` invocations + a `playbooks/` directory of JSON configs, we get the orchestration without the project-graph baggage. **No `nx.json` at repo root.**
- **@turbo/gen** is Turborepo-native, Plop-based, and Handlebars-driven. It owns the file-emission layer (`turbo gen run <name>`). Templates live under `tools/codegen/templates/` (turbo/generators/config.ts sets `TEMPLATES_ROOT = resolve(moduleDir, '../../templates')`).
- **Custom Bun scripts** handle the operations neither tool offers natively: AppModule wiring (ts-morph), DOC-GRAPH refresh, ai/curaos mirror creation, lefthook install, Verdaccio `.npmrc` injection, ai-rule cross-checks.

### Forbidden alternatives (per D1)

- Pure Bun + `@clack/prompts` only (lacks playbook composition for 3-layer service emission).
- EJS templates (mismatches @turbo/gen Handlebars internals; would force a parallel template engine).
- Nx as build system (`nx.json` with build/test targets) - violates `ai/rules/curaos_bun_primary_rule.md` + `ai/rules/curaos_speed_patterns_rule.md` (Turborepo canonical task runner).

## AST mutation strategy (M6-S3 - ts-morph)

Two ts-morph 23.x passes run per layer after `@turbo/gen` emits the file tree:

1. **`wireAppModule(serviceDir, { moduleName, importPath })`** - opens
   `<serviceDir>/src/app.module.ts` via a `ts-morph` `Project`, locates the
   `@Module({ imports: [...] })` decorator on the exported `AppModule` class,
   and idempotently adds:
   - The `import { ModuleName } from '<path>'` statement (merged into an
     existing import with the same specifier when possible).
   - A bare identifier reference inside the `imports: [...]` array.
2. **`emitBarrel(serviceDir, exports[])`** - opens or creates
   `<serviceDir>/src/index.ts`. For each requested export, ensures a
   `export { Name } from '<spec>'` declaration exists exactly once. Existing
   declarations with the same specifier accept additional named members.

Both passes are idempotent (D2 binding: "re-running generator must NOT
duplicate imports / decorator entries"). Detection relies on AST inspection,
not text matching, so reformatting / re-ordering does not confuse the
second-run no-op path.

### Anchor comments

The generated `src/app.module.ts.hbs` carries two anchor comments:

- `// @curaos-codegen:auto-wire-import-statements` - above the existing
  imports block. New `import` statements land after this anchor.
- `// @curaos-codegen:auto-wire-imports` - inside the `imports: [...]` array.
  New module references land after this anchor.

Anchors are advisory: `wireAppModule` still works when they are absent
(falls back to "append after last existing import" / "append to end of
array"). Their main job is to keep the generated file readable when many
modules are wired over time - new entries cluster after the anchor instead
of randomly intermixing with hand-edits.

### Why ts-morph (not text replace)

- Survives reformatting: Biome / Prettier can reflow the file without
  breaking the second-run no-op path. A `sed`/`sd`-style replace would
  fight any whitespace change.
- Survives `forRoot()` wrapping: `ConfigModule.forRoot()` is detected as
  the same registration as a bare `ConfigModule` reference, so the
  generator never double-registers a hand-wrapped module.
- Cheap: the source files are tiny (<30 LoC), so ts-morph startup
  dominates wall time - ~150 ms per file on M-series hardware. Acceptable
  for a generator that runs interactively.

### Live emission gating

S3 ships the ts-morph code paths AND wires them into both:
- The `--dry-run` orchestrator output (`src/index.ts` prints planned
  mutations alongside the @turbo/gen file plan).
- The @turbo/gen action chain (`turbo/generators/config.ts` registers
  `wireAppModule` + `emitBarrel` as custom Plop actions appended after
  each layer's `addMany`).

Real emission (writing to disk) stays gated to M6-S7 close-gate, matching
the rest of the harness. Until then, dry-run is the recommended path -
the ts-morph code is exercised exclusively by unit tests against in-memory
sources.
