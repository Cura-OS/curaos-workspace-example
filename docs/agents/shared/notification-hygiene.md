# Shared: PR Inbox-Notification Hygiene

Canonical fail-closed inbox rules for both [prompts](../milestone-orchestration-prompt.md) (worker side: [one-task prompt](../one-task-execution-prompt.md) section 9). A merged PR leaves a `github.com/notifications` entry that NOTHING removes on its own; three escape routes leak entries into the user's inbox (workflow-merged-without-clear, directly-merged, and the post-merge `state_change` re-notification GitHub emits after close).

- **Whole-inbox converger (the wave gate):** `bash "$WORKSPACE_ROOT/scripts/sweep-pr-notifications"` (dry-run shows would-CLEAR vs HELD) / `--apply`. It walks every PR notification, applies the `safe-to-clear-notification` predicate (terminal state + zero unresolved review threads + no `needs-human` thread), CLEARs the passes, and HOLDs + reports the rest. Fail-closed, idempotent, cron-safe; it catches all three escape routes because it scans the whole inbox, not just the PRs this session merged. Run it after every PR-merge wave; do NOT rely on remembering per-PR clears.
- **Per-PR escape hatch:** `bash "$WORKSPACE_ROOT/scripts/pr-notification-gate" --apply OWNER/REPO N` re-checks the same predicate and clears only when safe (exit 0); BLOCKs otherwise (exit 2 = PR still open, exit 3 = live finding or `needs-human`). It wraps the lower-level `mark-pr-notification-done` primitive (exact `/repos/OWNER/REPO/pulls/N` match, never substring, no re-check). The wave gate remains the whole-inbox sweep, not a per-PR loop.
- **Capture-then-resolve (binding):** on exit 3 or a HELD merged PR, either (a) the finding is already addressed: resolve the thread, or (b) it is genuinely live on `main`: file a follow-up issue FIRST so the finding is not lost, annotate/resolve the thread referencing that issue, then re-run. NEVER clear a notification whose finding is both live and uncaptured. Outdated/moot threads (`isOutdated=true`, code rewritten by a later PR) resolve as moot or pass `--allow-unresolved`. Codegen-template findings file against the codegen module (a template defect replicates to every generated service, per [[curaos-generator-evolution-rule]]).
- **Leave `needs-human` notifications in place.** Report `NOTIFICATION: <cleared|left-needs-human|held-awaiting-graphql-thread-check|n/a>`.
- **Quota-safe:** notification enumeration + mark-read are REST-supported and keep running when GraphQL is low. Prove the zero-thread case from REST PR state + review/review-comment reads first; use targeted GraphQL only for `reviewThreads.isResolved` / `resolveReviewThread`. A quota miss HOLDs that PR only (`NOTIFICATION: held-awaiting-graphql-thread-check`) and does not stop the sweep from clearing unrelated safe notifications.
- **Gate-bypass hazard (the dominant leak cause):** a PR merged directly (`gh pr merge`, the GitHub UI; tell-tale: `reviewDecision` empty / `mergedBy` = the human account, no gate run) never resolves its review threads, so findings strand unresolved on a merged PR and the sweep correctly HOLDs forever. Server-side branch protection requiring resolved conversations is the proper fix but needs a paid plan on private repos; until then enforcement is procedural: (1) prefer merging through `pr-verify-merge` so threads are resolved-or-captured as part of the merge; (2) after ANY out-of-band merge, immediately run the sweep and capture-then-resolve every HELD live finding, THEN re-sweep. A wave is not done while any merged PR holds an uncaptured live finding.
- **Last-action ordering (RP-18, binding):** the final pre-STOP `sweep-pr-notifications` dry-run runs as the LAST action of the session, after at least a final sweep after merge gates complete. Sequence: last merge -> thread sweep -> apply sweep -> final dry-run -> terminal report. Nothing (memory writes, HANDOVER refresh, closeout comments) may follow the final dry-run except the terminal report itself. Do NOT add a daemon for this (user directive, AUTO-DECISION-LOG row 53).
- **Wave-done precondition:** a clean dry-run showing only HELD open / `needs-human` entries, i.e. the inbox holds zero clearable PR notifications.
