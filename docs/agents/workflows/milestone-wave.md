# milestone-wave

ONE non-interactive pass of the milestone-orchestration Goal Setter: discover candidates â†’ triage gate â†’ break down oversized issues â†’ **prioritize by unblock-leverage + partition safe lanes (`wave-prioritize`)** â†’ dispatch `task-execute` per ready lane in leverage order â†’ `pr-verify-merge` per PR â†’ **foresight-sweep (seed staged future/dependency work)** â†’ report. Composes `pm-triage-gate` + `wave-prioritize` + `breakdown` + `task-execute` + `pr-verify-merge` + `foresight-sweep`.

The **Prioritize** phase runs [`wave-prioritize`](wave-prioritize.md) over the ready leaves: it ranks them by how much downstream work each transitively unblocks (deterministic dep-graph math in [`scripts/lib/dep-graph.js`](../../../scripts/lib/dep-graph.js)), bundles same-owner compatible leaves into one lane, and partitions owner-root lanes into parallel-safe dispatch work, so the wave spends its first lanes on the keystones that open the most future parallel width. Leverage decides **order only** for the gates bound upstream (Â§3.4 triage, Â§3.5 research, Â§3.7 grill, ADR/spec acceptance); a high-leverage issue never skips a gate to dispatch sooner. The in-flight generator/SDK barrier is NOT bound upstream of the partition: the wave executor enforces it deterministically right before prioritize (see Phases item 4 and Gates).

The **Foresight** phase runs [`foresight-sweep`](foresight-sweep.md) (wave mode) after verify: it discovers future work or dependency work introduced or implied by this pass (debt, deferred decisions, next-version prereqs, research gaps) and seeds each as a staged `needs-triage` + `foresight` issue (Target Version set when known, Backlog at birth) via `foresight-capture`'s handoffâ†’focused-subagentâ†’issue chain. Capture itself starts no implementation work and is advisory/non-blocking (a foresight failure never fails the wave). A later all-open Â§3.4 triage pass may promote relevant, complete, unblocked foresight work to `ready-for-agent` while preserving the marker. Cross-version deep foresight is a separate on-demand/scheduled `foresight-sweep mode=cross-version` run.

## Why this is ONE pass, not the whole Goal Setter

A full wave is **orchestrator-chained** - it spans turns, escalates to the user (Â§3.6), and applies judgment between phases. A workflow cannot do those (single session, no mid-run `AskUserQuestion`). So `milestone-wave` does the **deterministic legs of one pass**, **auto-applies any decision point that carries a clear recommendation** (logging each choice to `ai/curaos/docs/adr/AUTO-DECISION-LOG.md` per [[curaos-recommendation-auto-apply-rule]]), and **stops + reports** only what genuinely requires the orchestrator/user (no-recommendation trade-offs, irreversible/destructive/T3 actions, unapproved-scope expansions) in `needs_user`. The orchestrator (the [milestone-orchestration-prompt](../milestone-orchestration-prompt.md) main loop) re-invokes it across turns, handling `needs_user` between passes. It is NOT a replacement for the prompt - it is the prompt's per-pass executor.

## When to invoke

- The orchestrator (or you, via `/milestone-wave`) wants to advance all open issue work one deterministic pass.
- Invoke the strongest available orchestration path in the active harness, not a hardcoded foreign runner. Claude Code uses `Workflow({ scriptPath: "scripts/workflows/milestone-wave.workflow.js", args: { milestone: "active" } })`; Agent Workflow Kit capable harnesses use `agent-workflow-kit workflow-run milestone-wave --args-json '{"milestone":"active"}' --json`; Hermes uses native `todo` for phases, `delegate_task` for isolated lanes/reviews, and terminal/file tools for deterministic gates; Codex uses the configured adapter path; other harnesses follow this playbook directly. Manual execution is fallback only after an executor outage, `needs_user`, or `workflow-defect`.
- `dry_run=true` first to see the plan (what's ready, what'd dispatch, what's blocked) before mutating anything.
- **Real-agent mode (kit CLI only; RP-44):** the kit CLI defaults to STUB agents (`schemaDefaultAgent` - control-flow/plan-only; a stub pass is not evidence the wave ran). Append `--real-agents` (optionally `--agent-timeout-ms <ms>`, valid only with the flag) to spawn real CLI agents (`claude -p` / `codex exec` per agentType): `agent-workflow-kit workflow-run milestone-wave --args-json '{"milestone":"active","dry_run":true}' --real-agents --json`. Claude Code's native `Workflow` tool always runs real agents. Do not use `--max-concurrent-agents` as a throughput throttle; it is a hard cap that rejects extra calls. Use `triage_batch_size` for the queued triage throttle if provider limits require a smaller batch. Alternate harness use is explicit only: the wave never silently falls back from Codex to Claude. If Codex quota/credits are unavailable and immediate real agents are required, rerun with an intentional same-tier configured harness selection such as `--default-agent-type claude` plus a matching `dispatch_agent_type`, and record that choice in the run notes. A DISPATCHING real-agent run requires the in-flight generator/SDK/contracts preflight first (paginated probe empty, output archived with the run evidence); when the executor-side barrier cannot be proven active, constrain the run to `dry_run: true` (no-dispatch mode). See the orchestration prompt's "Real-agent mode" note.
- Before any non-dry broad run, the orchestrator must have user approval for either a ready-open-issues wave or an unblock-prep wave, then create/find a main local issue in `.scratch/state/symphony-work/local-issues.sqlite`. Each scan, triage, blocker, dispatch, verification, and follow-up row produced by the pass records `parent_id` to that main issue when practical. GitHub remains the tracker adapter; the local issue DB is the wave's machine progress spine.

## Inputs

`milestone`: legacy selector; `active` (default) scans every open org issue. `max_lanes` is an OPTIONAL planning throttle - default UNCAPPED for lane planning. `triage_batch_size` is the queued real-agent triage throttle (default 3). `submodule_jobs` controls wave and lane submodule update parallelism (default 8). `dry_run` plans only. `auto_merge` lets the PR gate merge green PRs.

## Phases

0. **Wave setup (P5c, issue #706; opt-in `init_submodules: true`)** - before Scan, run `git submodule update --init --recursive` (deterministic git via `scripts/lib/workflow-git.js initSubmodulesRecursive`, not an agent prompt) so pre-push typecheck hooks pass on uninitialized gitlinks instead of being masked with `--no-verify`, then re-check two-level pointer drift via `git submodule status --recursive` (`-` not initialized, `+` checked-out != pinned in the superproject index, `U` conflict). Logged, fail-soft (never fails the wave), and skipped under `dry_run`. Default OFF so the contract-test harnesses never trigger live recursive init; real dispatched waves pass `init_submodules: true`.
   Wave setup also records the selected runner and local main issue id before scan. If the harness cannot run the committed executor, native fallback keeps the same phase names and local child issue evidence instead of creating a separate untracked plan.
1. **Scan + select** - query every open org issue, read Project metadata (`Target Version`, `CuraOS Milestone`, Status), and build the full candidate set. For `milestone: "active"`, the workflow first reads the `gh-budget` GraphQL quota snapshot; when remaining quota is 0 it returns `terminal: "blocked-by-external"` with `kind: github-graphql-quota` before starting `milestone-active-scan`, so a known quota outage never burns a scan or triage batch. Otherwise it runs the deterministic `milestone-active-scan` helper against Project #2 and REST issue search, then skips the old whole-universe scan agent. The deterministic helper is authoritative for `milestones`, `candidates`, `paper_blocked_candidates`, `dependency_cleared`, `open_prs`, and `generator_inflight`; the next **Triage gate** phase owns paper-vs-real classification in bounded per-issue real-agent batches. This avoids one large model call timing out or dropping the all-open tracker set. `CuraOS Milestone` is retained as grouping metadata only; it does not narrow candidates or block dispatch. The active helper strips `GITHUB_TOKEN`, retries transient GitHub 5xx/504/unicorn failures, and after bounded retries returns `blocked_by_external` (`kind: github-project-api-transient`) instead of letting any LLM invent an empty set. Non-active legacy selectors may still use the scan agent. For each blocker or decision point found in triage: (i) check whether a clear recommendation exists (research points one way, one option is the obvious low-blast-radius default); if yes, **auto-apply it** per [[curaos-recommendation-auto-apply-rule]] and record the choice in `ai/curaos/docs/adr/AUTO-DECISION-LOG.md` with marker `(auto-applied per recommendation, 2026-05-29 directive)` - do NOT surface to `needs_user`; (ii) only surface to `needs_user` when NO recommendation exists (genuine trade-off), the action is irreversible/destructive/T3-class, or the scope is unapproved. Do NOT guess a recommendation where none exists.
   - **`dependency_cleared` bucket (Scan, in code).** The deterministic `milestone-active-scan` helper emits a `dependency_cleared` bucket: every `foresight`/`blocked` issue whose `blocked-by:` frontmatter names refs that are now ALL CLOSED (strict `owner/repo#N` validation; fail-closed on a transient probe failure). A merged blocker is the most common gate-opener, so the wave folds this bucket into `triage_pool` alongside all issue candidates and Â§3 promotes the just-unblocked story the same pass its blocker merged. A label-only scan misses these (the story keeps its `foresight`/Backlog marker until triage promotes it); this is the M16 #538 -> M17-S2 #545 class. The orchestrator MUST re-run the scan after every merge so a fresh `dependency_cleared` bucket is consumed before the next dispatch (user directives 2026-06-08 and 2026-06-17: always auto-pick every unblockable or relevant foresight dependency task; do not hand-notice a cleared gate).
2. **Triage gate** - `pm-triage-gate` over the surviving candidates (per-candidate triage, then Project sync + wire only for dispatch-eligible rows, then mirror). The wave creates one deterministic Project item-list cache before candidate sync and passes it through `project_items_cache`; it must not run one full `gh project item-list` per candidate. That cache is the SHARED TTL board snapshot (RP-38: `scripts/lib/gh-project.js boardSnapshot()`, one file at `.scratch/workflow-cache/roadmap-items.json`, 5-min TTL, truncation fail-closed) instead of a unique timestamped file per pass; within the TTL a read costs zero network calls, and any mutating sweep invalidates it. If that Project cache cannot be read because GitHub GraphQL quota is exhausted, the wave logs `PROJECT-CACHE-DEGRADED`, continues issue triage, and skips Project metadata sync for otherwise dispatchable rows; Project fields are metadata and must not starve label triage or independent ready lanes. **Batch prefetch (RP-36):** before each queued triage batch the wave reads the local Codex runtime status-line/session telemetry when the active harness is Codex. If fresh evidence says the Codex runtime has no credits or hit a provider limit, the wave logs `TRIAGE-RUNTIME-PREFLIGHT`, returns `terminal: "blocked-by-external"` with `agent-runtime-quota`, and does not launch the batch. Then it reads that batch in <=2 GraphQL calls per 100 issues via `gh-project.js batchIssueRead` (one aliased document) and threads each record into `gh-issue-triage` as `prefetch` (and its body onward into the RP-39 prompt threading). `gh-issue-triage` consumes that prefetched body, labels, and native parent as authoritative input instead of issuing its own mandatory `gh issue view`; for complete, already-labelled, unblocked rows it may apply the deterministic fast path without spending a model turn. A non-quota batch failure degrades only that batch to per-issue deterministic reads. A GraphQL quota failure logs `TRIAGE-QUOTA-DEFERRED`, stops launching later batches, and records every unlaunched candidate in `not_ready` with `missing=github-graphql-quota` so completed ready lanes can continue without burning more agent turns. Non-dispatchable rows keep the REST-applied issue label state and are reported in `not_ready`; the wave does not spend GraphQL Project mutations or native issue-tree writes on rows that cannot dispatch. The inline gate threads deterministic execution Status with triage fields (`ready-for-agent|ready-for-human -> Ready`, blocked or real blocker -> Blocked, triage/info/future-version-only -> Backlog). A `foresight` marker alone does not force Backlog. `gh-project-sync` returns `milestone: NONE` for unmapped/skipped milestone writes as metadata only; this does not block readiness. `gh-subissue-wire` may return `subissues_depth_limited` when GitHub's native issue tree cannot represent a deeper edge; that explicit exception is accepted only with the already-curated frontmatter/project parent evidence. Only candidates that satisfy the full readiness predicate proceed.
   Runtime-held rows (`agent-claimed:*` / `agent-PR-open`) stay out of dispatch but are returned in `pending_tracker_work` with `route: "runtime-lane-check"`, so a stale claim or stale PR-open label cannot hide older work from the orchestrator.
   `pending_tracker_work` rows are reported continuation work, not a global dispatch freeze. `runtime-lane-check`, `investigate`, `user-escalation`, `tracker-triage`, `tracker-repair`, `planning-breakdown`, `blocker-follow-up`, `sync-degradation`, `triage-retry`, and `closed-out` stay visible for cleanup or follow-up while independent ready lanes continue through breakdown, prioritize, and dispatch. RP-48 pre-breakdown remains allowed for `dependency_cleared` rows that still need decomposition because it stages child specs; later Â§3.4 triage can promote relevant ready children.
   Dropped triage pipeline rows are terminal external blockers: if the runtime returns `null` for any candidate (for example provider rate limiting or a child workflow failure), the wave returns `terminal: "blocked-by-external"` with `kind: agent-triage-pipeline-dropped` and refuses partial all-open issue accounting. True provider triage outages such as `agent-runtime-quota` remain terminal. GitHub GraphQL quota before active scan is terminal because tracker discovery cannot be trusted; GitHub GraphQL quota after queued triage has already completed some rows is deferred as `not_ready` rows so independent ready lanes can continue. Per-candidate triage child failures, including `agent-runtime-unavailable` timeouts and `triage-workflow-failed`, degrade to `not_ready` with route `triage-retry`, so independent ready lanes can continue until `agent-runtime-unavailable` exceeds `triageRuntimeUnavailableLimit`; at that point the wave logs `TRIAGE-RUNTIME-UNAVAILABLE`, returns terminal `agent-runtime-unavailable`, and stops launching more child agents. A rate-limited triage fan-out is never completion evidence.
3. **Breakdown** - any ready candidate that is not grab-able â†’ `breakdown` (recursive split). New leaves re-enter the ready set. Before invoking the read-only assessor or issue-creating split agent, the executor reads native subissues for the candidate. If the parent already has open child issues, hold the parent in `not_ready` with route `breakdown-existing-children`; those children are picked up by the all-open issue scan instead of creating another decomposition set. The assess prompt receives the RP-36 batch record's body injected as the AUTHORITATIVE prefetched issue body (RP-39: do not re-fetch; ONE comments spot-check permitted, not mandated), falling back to the mandated `gh issue view` read when no record exists. If the assessor says `grabable:false` but returns no `proposed_children`, or if the read-only assessor times out, that is inconclusive model output, not a valid split; keep the original ready candidate as a dispatchable leaf and let downstream gates block with real evidence if it is actually too broad. If the issue-creating split agent times out after the assessor proved the parent is too broad, hold the original parent in `not_ready` with route `breakdown-retry`; independent ready lanes still continue through prioritize and dispatch. If GitHub rejects breakdown child wiring because the native sub-issue tree would exceed 7 layers, log `WIRE-DEPTH-LIMIT`, record the children as `subissues_depth_limited`, and continue; GitHub's display-depth cap must not abort the whole wave.
3.5. **Pre-breakdown trigger (RP-48)** - every `dependency_cleared` issue that triage did NOT promote into the active ready set may still be an Epic/Story that needs decomposition before workers can grab it. For each (capped per pass by the calibration throughput sizing signal `analyze().sizing.suggestedWaveSize`, fallback 8), a read-only draft agent proposes its vertical tracer-bullet story set; non-empty drafts are filed through `foresight-capture` as staged foresight work with Project metadata, so the next Â§3.4 pass can promote relevant, complete, unblocked children instead of commissioning a prep session. Reported in `pre_breakdown` `{ drafted, skipped }`; advisory and fail-soft (a drafting failure never fails the wave); `dry_run` drafts without seeding.
4. **Partition** (in code) - compute ALL safe parallel lanes (no shared git working tree: submodule checkout / parent `curaos` repo / `.gitmodules` / lockfile / migration / CI / doc-graph). Same-owned-root ready leaves are bundled into one lane (`issues[]`) before any new agent is added. UNCAPPED by default - emit every collision-free owner-root lane; truncate only if a finite `max_lanes` was passed. **Gen-evo barrier (deterministic, RP-04):** when the deterministic scan reports `generator_inflight` (an `agent-claimed:*`/`agent-PR-open` codegen/SDK/contracts lane), generated-scope candidates (the scan's `isGeneratorScope` test over ref + module/owned-path frontmatter) are held out of the ready leaves BEFORE prioritize, so they never reach dispatch lanes; each held entry carries reason `gen-evo barrier: <ref> in-flight`. `context-load` inside dispatch remains the per-lane LLM backstop. Candidates are built deterministically from the triage context (no LLM enrichment step; RP-47 removed the haiku enrich-frontmatter agent); `wave-prioritize` itself backfills any missing `priority`/`effort` from issue frontmatter, so the calibration record captures real values. Never pass an empty array to `wave-prioritize` unless there are truly no ready leaves. **Scheduling policy (RP-51):** the wave derives `near_completion` refs deterministically from the shared board snapshot (items at Status In Progress / In Review = resumed work or an open PR awaiting verify) and passes them to `wave-prioritize`, whose lane order finishes near-completion lanes FIRST, then critical path deepest-first, FIFO within a priority class; the serial dispatch loop consumes lanes in that order. A `degraded:true` prioritize result (RP-46 edge-fetch failures) is logged (`PRIORITIZE-DEGRADED`); ranking stays usable and the calibration append is skipped by the lib.
5. **Dispatch** - default dispatch is worktree-isolated: each collision-safe lane gets a branch created from the repository remote default ref in its own `.worktrees/<lane>` checkout through `createIsolatedLaneWorktree`. This keeps branch-changing implementation lanes parallel-safe and prevents another session from stashing or reverting the integration checkout. Explicit low-resource same-checkout fallback serializes actual branch-changing dispatch from workflow `ROOT`; that compatibility mode must restore or stash the shared checkout back to the default branch after every lane. Branch creation failure returns `blocked` with `branch-create-failed`, and no `tdd-implement`/PR phase runs. A lane may open a PR only when `tdd-implement` returns a non-empty Section 8.1 `verification_evidence` block, the PR agent returns a valid `owner/repo#N` PR ref, and post-PR default-branch restore succeeds; missing evidence, empty/malformed PR ref, or failed restore is `blocked`, never `pr-open`. If a lane returns `workflow_defect_kind:"tdd-implement-no-op-done"`, the same-checkout fallback halts the remaining serial attempts and records those lanes as blocked by the same barrier instead of creating more zero-work branches.
   Worktree dispatch propagates the child `task-execute` result payload (`pr-open`, `blocked`, `needs-user`, or `errored`), not the wrapper workflow-run status. A wrapper `completed` without a task result is not lane completion evidence. Lane-level `needs-user` records are copied into the wave `needs_user` output with `kind: "Worker-needs-user"` so the orchestrator sees the concrete decision.
   Inline `tdd-implement` keeps harness-aware model routing: logical tiers (`opus`/`sonnet`/`haiku`) are omitted as raw model ids by default unless the runner explicitly opts in with `AGENT_WORKFLOW_KIT_PASS_LOGICAL_MODELS=1`. A schema-default `done` from unsupported tier-name passthrough is preserved as `workflow_defect:true` with `workflow_defect_kind`, not accepted as dispatch success.
   The branch/restore/stash/collision/PR-ref helper surface is single-owned in `scripts/lib/workflow-git.js`; `milestone-wave` and `task-execute` must both load that helper rather than maintaining duplicated git helper blocks. Likewise (RP-20) the triage-status derivation (`statusFromTriage`/`projectFieldsForSync`) and the merge+hygiene deterministic core (`ghPrCommand`/`isBlockedHarnessUnavailable`/`grillShaMismatch`) are single-owned in `scripts/lib/triage-status.js` + `scripts/lib/merge-hygiene.js`: this executor imports them directly; the Claude-style executors (`pm-triage-gate`, `pr-verify-merge`) keep byte-identical inline copies pinned by truth-contract equality tests.
   Remote branch probes fail closed: only `git ls-remote --exit-code` status `2` means "no matching remote branch"; transport/auth/network/default-branch-resolution failures block branch creation and must not fall back to `main`.
   **Deploy-lane credential preflight (RP-52):** a lane whose issue/module/owned-path marks image build/publish/sign work (the #588 GHCR class) runs `scripts/preflight-credentials --registry-probe` BEFORE dispatch (deterministic, fail closed: exit 65 = missing token scope, 66 = registry write rejected, 70 = cannot prove). Any nonzero exit blocks ONLY that lane with the exact operator unblock one-liner (`gh auth refresh -h github.com -s write:packages,read:packages && scripts/preflight-credentials`) so the failure surfaces in the operator queue instead of hours into an image build.
   **Budget preflight (RP-42):** before fan-out the executor reads the rate-limit ledger (`scripts/lib/gh-budget.js readBudgets()`, persisted at `.cache/gh-budget-ledger.json`) and defers the entire fan-out into `held` (reason reported, `budget-preflight deferral: ...`) when any of the three budgets (core REST / GraphQL / search) is below 2x the estimated wave cost after the ~500-GraphQL-point closeout-sweep reserve; GraphQL starvation with core headroom downgrades enumeration to REST instead of deferring. A broken budget probe is fail-open (logged, never strands a dispatchable wave).

## Why this composite INLINES its children (not workflow()-nested)

The Workflow runtime caps `workflow()` nesting at ONE level (`agent()` nests freely). `milestone-wave` is the TOP workflow, so it CANNOT `workflow()`-call the child composites (`pm-triage-gate`, `breakdown`, `task-execute`, `pr-verify-merge`) - each of those itself calls `workflow()` on its atomics, which would make the wave 2 levels and throw. Instead the wave **inlines** each child's body directly, calling the underlying ATOMICS (`gh-issue-triage`, `gh-project-sync`, `gh-subissue-wire`, `gh-roadmap-mirror`, `context-load`, `tdd-implement`, `lens-review`, `opposite-harness-grill`) one level deep - legal. `composition: inline` in the contract records this; `composes:` keeps the real lineage for the library index. Each child composite stays **independently runnable** as a TOP via its own `scriptPath` (where its own `workflow()`-to-atomic calls are depth 1, legal). The inline copies in the wave and the standalone child bodies are kept in sync by hand - same atomics, same schemas; if a child's prompt/schema changes, update the wave's inline copy too (the sync gate compares CONTRACT blocks, not bodies).
